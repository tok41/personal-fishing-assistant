# 釣行アシスタントアプリケーション - 要件定義書 v1

## 概要

過去の釣行記録を知識ベースとして活用し、ユーザーの相談内容に応じて関連情報を自動検索・参照しながらAIと対話できる釣行支援アプリケーション。

## 背景と目的

### 解決したい課題

- **現状の問題点**
  - 釣行記録をNotionで管理しているが、AI（ChatGPT/Claude Web版）に相談する際、毎回手動で記録を貼り付ける必要がある
  - 過去の記録を参照しながら相談したいため、チャットスレッドを分けたくないが、スレッドが長くなりすぎる問題がある
  - スレッドの要約を作成して新しいスレッドで再開する運用をしているが、要約により相談内容が薄れる懸念がある
  - 長いチャットを続けると、過去の内容がコンテキストから外れてしまう

### 目的

- 記録の本体はNotionなどに保持しつつ、相談内容に合わせて適切な過去記録を自動検索し、それを基に対話を進められる仕組みを提供する
- 長期的な対話履歴を適切に管理し、過去の相談内容も含めた文脈を保ちながら対話できる環境を構築する

### 想定する利用者と利用シーン

- **利用者**: 個人（アプリケーション開発者本人）
- **利用シーン**:
  1. 釣行前の次回構想段階での相談
  2. 釣行直前の具体的なプランニング
  3. 釣行後の記録整理と反省
  4. 一般的な釣りに関する知識の相談
  5. ぼやきや感想の共有

## スコープ

### 対象範囲（Phase 1: MVP）

- **釣行記録の管理**
  - NotionからエクスポートしたMarkdownファイルのインポート機能
  - 釣行記録のベクトルデータベース化（FAISS使用）
  - 釣行記録の全文ベクトル検索

- **基本的なAI対話機能**
  - ユーザーの質問に対して、関連する釣行記録を自動検索
  - 検索された記録を参照してAI（OpenAI API）が回答生成
  - シンプルなチャットインターフェース（Web UI）

- **技術基盤**
  - ローカル環境で動作するWebアプリケーション
  - Pythonベースのバックエンド
  - Reactベースのフロントエンド（または代替UI）

### 対象範囲（Phase 2: 実用レベル）

- **会話履歴管理機能**
  - 直近N往復の会話履歴の保持
  - 会話量に応じた自動要約機能（ユーザー確認あり）
  - 要約後の元会話ログの削除

- **コンテキスト情報の管理**
  - **プロフィール**: ユーザーの好みや特徴（よく行く場所、好きな釣法、所有道具など）
  - **釣行経験のサマリ**: 場所ごとの釣行回数、釣果の統計など
  - **最近の課題のまとめ**: 現在取り組んでいる技術課題や改善点

- **統合された回答生成**
  - 直近の会話履歴（フルテキスト）
  - プロフィール、サマリ、課題（要約情報）
  - 検索された釣行記録（RAG）
  - 上記すべてを参照した回答生成

- **初期プロフィール生成**
  - 最初の数回の会話から自動的にプロフィール・サマリ・課題を生成

### 対象外（Phase 3以降で検討）

- Notion APIを使った直接連携
- 釣行記録の自動入力支援
- Web検索機能との統合（Tavily API等）
- トピック切り替えの自動検知
- AI APIの切り替え機能（OpenAI ⇔ Claude）
- 構造化データ（タグ、釣法など）によるフィルタリング検索
- Notionへのプロフィール・サマリ・課題・会話履歴の同期
- 複数ユーザー対応
- 外部公開・クラウドデプロイ

## 機能要件

### FR-1: 釣行記録のインポート（Phase 1）

- **概要**: NotionからエクスポートしたMarkdownファイルを読み込み、システムに登録する
- **振る舞い**:
  - `data/records/`ディレクトリ内のMarkdownファイルを一括インポート
  - 各記録をパースしてテキストデータとして保存
  - ベクトル化してFAISSインデックスに登録
- **入力**: なし（`data/records/`固定）
- **出力**: インポート成功/失敗の通知、登録件数
- **前提条件**:
  - Markdownファイルが特定のフォーマット（Notionエクスポート形式）に従っている
  - ファイル名形式: `YYYY-MM-DD_場所名.md`（例: `2026-02-15_日立港.md`）
  - ディレクトリ構造: フラット（サブディレクトリは探索しない）
  - 対象拡張子: `.md`のみ
- **制約**:
  - Phase 1では構造化データ（タグ、釣法など）の抽出は行わない
  - 全文をテキストとして扱う
- **異常系**:
  - インポート対象が0件の場合: 「0件」と理由を表示して処理終了
  - Markdown形式不正のファイル: スキップしてエラーログに記録、他ファイルの処理は継続

### FR-2: 釣行記録の検索（Phase 1）

- **概要**: ユーザーの質問内容から関連する釣行記録を検索する
- **振る舞い**:
  - ユーザーの質問文をベクトル化
  - FAISSを使って類似度の高い記録を取得（上位5件）
  - 検索結果はユーザーに提示せず、内部的にAIに渡す
- **入力**: ユーザーの質問文
- **出力**: 関連する釣行記録のリスト（内部データ）
- **前提条件**: 釣行記録がインポート済み
- **制約**:
  - Phase 1では全文ベクトル検索のみ
  - フィルタリング機能なし
- **異常系**:
  - 登録済み記録が0件の場合: 空のリストを返す（FR-3でフォールバック）
  - 検索結果が0件の場合: 空のリストを返す（FR-3でフォールバック）

### FR-3: AI対話機能（Phase 1）

- **概要**: ユーザーの質問に対してAIが回答を生成する
- **振る舞い**:
  - ユーザーが質問を入力
  - システムが関連する釣行記録を検索（FR-2）
  - 検索された記録とユーザーの質問をOpenAI API（gpt-5-mini）に送信
  - AIの回答をユーザーに表示
  - 検索結果が0件の場合、「関連記録がない」旨を表示し、記録を参照せず一般的な回答を生成
- **入力**: ユーザーの質問文
- **出力**: AIの回答文
- **前提条件**:
  - OpenAI APIキーが設定済み
- **制約**:
  - Phase 1では会話履歴を保持しない（ステートレス）
  - 各質問は独立して処理される
- **異常系**:
  - OpenAI APIキー未設定時: アプリ起動時にエラーメッセージを表示し、AI対話機能を利用不可にする
  - OpenAI APIエラー時: エラーメッセージをユーザーに表示、再試行可能にする
  - タイムアウト（30秒）: タイムアウトメッセージを表示

### FR-4: チャットインターフェース（Phase 1）

- **概要**: ユーザーがAIと対話するためのWeb UI
- **振る舞い**:
  - ChatGPTライクなチャット画面
  - ユーザーが質問を入力
  - AIの回答がチャット形式で表示
  - 直近の会話履歴が画面に表示される（Phase 1では保存されない）
- **入力**: ユーザーの質問文
- **出力**: AIの回答、会話履歴の表示
- **前提条件**: ローカルサーバーが起動している
- **制約**:
  - Phase 1ではページリロードで会話がクリアされる
  - 会話の永続化なし

### FR-5: 会話履歴の保持（Phase 2）

- **概要**: 直近30往復の会話を保持し、対話の文脈を維持する
- **振る舞い**:
  - 会話が30往復を超えたら、システムが要約を提案
  - ユーザーが承認したら、古い会話を要約してプロフィール・サマリ・課題に反映
  - 要約後、元の会話ログは削除
  - 直近30往復の会話は常にフルテキストで保持
- **入力**: ユーザーとAIの会話
- **出力**: 保持された会話履歴、要約提案の通知
- **前提条件**: Phase 1の機能が実装済み
- **制約**:
  - トピックの自動検知は行わない（会話量ベース）
  - 要約にはユーザーの確認が必要

### FR-6: プロフィール・サマリ・課題の管理（Phase 2）

- **概要**: ユーザーの特徴、釣行経験、現在の課題を要約情報として管理
- **振る舞い**:
  - **プロフィール**: よく行く場所、好きな釣法、所有道具、釣りのスタイルなど
  - **釣行経験のサマリ**: 場所ごとの釣行回数、魚種ごとの釣果、時期ごとの傾向など
  - **最近の課題**: 技術的な課題（例: タナ取り、サビキの絡み対策）、改善点、次回試したいこと
  - これらは会話の要約時に自動生成・更新される
  - 初回起動時は空、数回の会話後に自動生成
- **入力**: 会話履歴の要約結果
- **出力**: プロフィール、サマリ、課題のデータ（JSON形式）
- **前提条件**: 一定量の会話が行われている
- **制約**:
  - 自動生成のため、精度は会話の内容に依存
  - ユーザーによる手動編集機能はPhase 3以降

### FR-7: 統合コンテキストでの回答生成（Phase 2）

- **概要**: 複数の情報源を統合してAIが回答を生成
- **振る舞い**:
  - ユーザーの質問に対して、以下を収集:
    1. 直近30往復の会話履歴（フルテキスト）
    2. プロフィール（要約情報）
    3. 釣行経験のサマリ（要約情報）
    4. 最近の課題（要約情報）
    5. 関連する釣行記録（RAG検索結果）
  - これらを統合してOpenAI APIに送信
  - AIが文脈を踏まえた回答を生成
- **入力**: ユーザーの質問
- **出力**: 統合コンテキストを考慮した回答
- **前提条件**: FR-5、FR-6が実装済み
- **制約**:
  - コンテキストの長さ制限に注意（トークン上限）
  - 必要に応じて情報を削減・要約

## 非機能要件

### パフォーマンス

- **応答時間**:
  - 釣行記録の検索: 1秒以内
  - AI回答生成: OpenAI APIのレスポンス時間に依存（通常5〜15秒）
  - インポート処理: 25件の記録で30秒以内
- **スループット**: 単一ユーザーのローカル利用のため、特に制約なし

### 可用性

- ローカル環境で動作
- AI機能の利用にはOpenAI APIへのアクセスが必要（インターネット接続必須）

### セキュリティ

- **APIキー管理**:
  - 環境変数または設定ファイル（`.env`）で管理
  - `.gitignore`に追加してリポジトリにコミットしない
- **データ保護**:
  - すべてのデータはローカルに保存
  - 外部への送信はOpenAI APIへのリクエストのみ

### 保守性

- **ログ**:
  - エラーログをファイルに出力
  - 検索クエリとAIへのリクエスト内容をデバッグログとして記録（オプション）
- **監視**: ローカル利用のため、特別な監視機能は不要
- **コード構成**:
  - Pythonバックエンドとフロントエンドを分離
  - 各機能をモジュール化して保守しやすく

### その他

- **拡張性**: Phase 3以降の機能追加を想定した設計
- **テスト性**: 各モジュールが独立してテスト可能な構成

## ユースケース

### UC-1: 釣行記録のインポート

1. ユーザーがNotionから釣行記録をMarkdown形式でエクスポート
2. エクスポートしたファイルを`data/records/`ディレクトリに配置
3. アプリケーションの「インポート」機能を実行
4. システムがMarkdownファイルを読み込み、ベクトル化してFAISSインデックスに登録
5. インポート完了の通知とともに、登録件数が表示される

### UC-2: 釣行前の計画相談（Phase 1）

1. ユーザーがチャット画面で「来週日立港に行く予定だけど、アドバイスほしい」と入力
2. システムが「日立港」に関連する過去の釣行記録を検索
3. 検索された記録（上位5件）とユーザーの質問をOpenAI APIに送信
4. AIが過去の記録を参照しながら、季節や潮汐、おすすめの釣法などをアドバイス
5. ユーザーが回答を確認し、必要に応じて追加質問（Phase 1では文脈は保持されない）

### UC-3: 釣行前の計画相談（Phase 2）

1. ユーザーがチャット画面で「来週日立港に行く予定だけど、アドバイスほしい」と入力
2. システムが以下を収集:
   - プロフィール（「サビキとルアーの両方を試したい人」）
   - 最近の課題（「タナ取りの精度向上」「サビキの絡み対策」）
   - 日立港に関連する過去の釣行記録
3. AIがこれらを統合して、具体的で個別化されたアドバイスを生成
4. ユーザーが「タナ取りのコツをもう少し詳しく教えて」と追加質問
5. システムが直近の会話を参照しながら、詳細な説明を生成

### UC-4: 釣行後の振り返り（Phase 2）

1. ユーザーが「今日は全然釣れなかった。風が強くて投げサビキが全然安定しなかったんだよね...」と入力
2. システムが「風が強い」「投げサビキ」に関連する過去の記録を検索
3. AIが過去の似た状況での対処法を提示しつつ、共感的な回答を生成
4. ユーザーが「次はどうしたらいいかな？」と質問
5. システムが最近の課題リストを参照し、具体的な次回対策を提案

### UC-5: 会話の要約（Phase 2）

1. ユーザーとAIの会話が30往復に達する
2. システムが「これまでの会話を要約して、プロフィールと課題リストを更新しますか？」と提案
3. ユーザーが「はい」を選択
4. システムが会話を要約し、プロフィール、サマリ、課題を更新
5. 元の会話ログは削除され、画面がリセットされる
6. 次回の質問から、更新された要約情報が参照される

### UC-6: 初回利用時のプロフィール生成（Phase 2）

1. ユーザーが初めてアプリを起動し、「釣りを始めて3ヶ月です。日立港でサビキをよくやっています」と入力
2. システムがこの情報を記録
3. 数回の会話を経て、システムが自動的にプロフィールを生成
4. 次回以降の質問で、このプロフィールが参照される

- [ ] 検索結果が0件の場合でも、一般的な知識に基づく回答が返されること
- [ ] OpenAI APIキー未設定時、起動時にエラーメッセージが表示されること
- [ ] OpenAI APIエラー時、エラーメッセージが表示されること

- [ ] ユーザーが要約を承認すると、プロフィール、サマリ、課題が更新されること
- [ ] 要約後、元の会話ログが削除されること

## 制約条件

### 技術的制約

- **プログラミング言語**: Pythonをベースとする（バックエンド）
- **ベクトルストア**: FAISSを使用（ローカル環境で動作）
- **AI API**: OpenAI APIを使用（gpt-5-mini）
- **実行環境**: ローカル環境（外部公開なし）
- **データ保存**: ローカルファイルシステム（`data/`ディレクトリ配下）

### リソース制約

- **開発体制**: 個人開発（開発者1名）
- **予算**: OpenAI APIの利用料金のみ（月額数千円程度を想定）
- **データ量**: 釣行記録25件からスタート、将来的に100件程度まで増加を想定

### 業務制約

- 個人利用目的のため、セキュリティや可用性の要求は緩やか
- 早期にMVPを動かし、段階的に機能追加する方針

## 依存関係

### 外部システム・API

- **OpenAI API**: gpt-5-miniの利用
  - APIキーの取得が必要
  - 利用料金が発生
- **Notion**: 現状の釣行記録管理に使用
  - Phase 1ではエクスポート機能を利用（API連携なし）

### 前提となる技術要素

- **Python**: 3.10以上
- **FAISS**: Facebookが開発したベクトル検索ライブラリ
- **フロントエンド**: React または Gradio/Streamlit
- **Webフレームワーク**: FastAPIまたはFlask（バックエンド）

### 機能間の依存関係

- FR-2（検索）は FR-1（インポート）に依存
- FR-3（AI対話）は FR-2（検索）に依存
- FR-5（会話履歴）は Phase 1の全機能に依存
- FR-6（プロフィール管理）は FR-5（会話履歴）に依存
- FR-7（統合コンテキスト）は FR-5、FR-6に依存
